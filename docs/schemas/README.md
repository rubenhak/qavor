# qavor — Manifest Schemas

Formal JSON Schema (draft 2020-12) for qavor's manifest model. Every manifest document is discriminated by a top-level `kind:` field, and one schema exists per kind. A master schema dispatches by `kind` so a single validator invocation can cover any document.

| File | Kind | Lives at |
|---|---|---|
| [qavor.schema.json](./qavor.schema.json) | _any_ (dispatcher) | apply to any qavor manifest document |
| [qavor.workspaces.schema.json](./qavor.workspaces.schema.json) | `workspaces` | multi-repo only: workspace root, generated by `qavor init` (single-repo projects have none) |
| [qavor.project.schema.json](./qavor.project.schema.json) | `project` | project repo root (multi-repo lists `repositories:`; a `standalone: true` single-repo project omits it) |
| [qavor.service.schema.json](./qavor.service.schema.json) | `service` | repo root or service sub-directory (also covers backing deps via `env.publish`) |
| [qavor.profile.schema.json](./qavor.profile.schema.json) | `profile` | anywhere; referenced by `profiles:` lists |
| [qavor.defs.schema.json](./qavor.defs.schema.json) | _shared $defs_ | referenced by `$ref` from every kind schema |

See [`docs/manifests.md`](../manifests.md) for narrative examples and [`docs/proposal.md`](../proposal.md) for the model overview.

## Multi-document YAML

A repo's manifests live in `qavor.yaml` at the repo root. The file may be a single document or a multi-document YAML (`---` separated). Repos with many services may instead use a `qavor/` directory of files. Both forms can mix kinds freely.

When validating multi-document YAML, validate each document against [qavor.schema.json](./qavor.schema.json), which dispatches to the appropriate per-kind schema based on the document's `kind`.

## Editor integration

VS Code / IntelliJ / Neovim YAML LSP can validate a single-kind manifest in-place by adding a schema header pointing at the matching per-kind schema:

```yaml
# yaml-language-server: $schema=https://qavor.dev/schemas/qavor.service.schema.json
kind: service
name: auth
```

For multi-document files, point the LSP at the master schema instead:

```yaml
# yaml-language-server: $schema=https://qavor.dev/schemas/qavor.schema.json
```

Until the schemas are hosted, point the LSP at the local file:

```yaml
# yaml-language-server: $schema=../path/to/docs/schemas/qavor.service.schema.json
```

## Compatibility & versioning

- `schemaVersion: 1` is the only supported version at v0; the field is optional and defaults to `1`.
- Backward-incompatible changes bump the integer; qavor refuses to load unknown versions.
- Forward-compatible additive changes (new optional fields) keep the version unchanged.

## Validation in CI

Until the qavor binary exists, manifests can be validated with any draft-2020-12 validator that resolves `$ref` across files, e.g.:

```bash
ajv validate \
  -s docs/schemas/qavor.schema.json \
  -r 'docs/schemas/qavor.*.schema.json' \
  -d 'path/to/qavor.yaml' \
  --spec=draft2020 --strict=false
```
