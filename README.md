# qavor

A CLI for managing a constellation of related repositories as one cohesive developer workspace.

> Status: **early design** — see [`docs/proposal.md`](./docs/proposal.md) for the full product proposal.

## Why

Modern product teams routinely span 5–50+ repositories: services, libraries, infra, data jobs, web apps. Onboarding and day-to-day work require chaining `git`, `docker compose`, multiple language toolchains, custom shell scripts, and tribal knowledge — and breaks the moment one repo is missing or misconfigured.

`qavor` is a single CLI that takes a developer from a fresh laptop to a fully running multi-service stack with one command, and then makes routine multi-repo operations — sync, status, commit, push, prepare, build, run, restart — first-class verbs.

## What it is (and isn't)

`qavor` is a **lean wrapper**. It owns:

- the declarative manifest model,
- the cross-repo dependency graph,
- environment composition with explicit precedence,
- and the orchestration loop.

It shells out for everything else: `git`, `docker` / `docker compose`, and language toolchains (`npm`/`pnpm`, `uv`/`pip`, `go`, `cargo`, …).

It is **not** a CI/CD system, a Kubernetes replacement, a devcontainer, or a new container runtime / package manager.

## Core concepts

Every YAML document carries a top-level `kind:` that selects its schema and orchestration semantics:

- `kind: workspaces` — pointer file at the workspace root (generated by `qavor init`).
- `kind: project` — at the seed repo root; defines the workspace and lists every repo in it.
- `kind: repo` — minimal per-repo metadata (groups, lifecycle hooks).
- `kind: service` — a runnable application, with `native` and `docker` runtime blocks.
- `kind: stateful` — an externally provided backing dep (postgres, kafka, redis, …), backed by `docker compose`.
- `kind: profile` — reusable runtime + env bundle, referenced by services and stateful manifests.

See [`docs/manifests.md`](./docs/manifests.md) for canonical examples and [`docs/schemas/`](./docs/schemas/) for the formal JSON Schemas.

## Typical workflow

```bash
qavor init <project-repo-url>   # clone seed repo, write workspaces pointer, clone everything else
qavor doctor                    # verify toolchains, container runtime, etc.
qavor prepare                   # install / sync dependencies across all repos
qavor up                        # start stateful deps + services in dependency order
qavor status                    # aggregated repo + service health
qavor logs <service>            # tailed, prefixed logs
qavor down                      # graceful shutdown
```

All multi-repo verbs accept selectors: `--repo`, `--group`, `--tag`, plus state filters like `--dirty`, `--ahead`, `--behind`.
